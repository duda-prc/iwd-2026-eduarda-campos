Boas práticas 3/3 · Estratégias de merge na sua branch main

Merge commit

> O que é: nasce um commit M só para fechar o PR — ele tem dois pais (ponta da main e ponta da feature). O trabalho do branch continua visível no grafo.
> Quando: o time gosta de ver cada integração como “caixote” no histórico e não se importa com mais ramificações no graph.
> Trade-off: o log fica mais ramificado; bisect e leitura linear podem exigir um passo a mais.

Antes do merge

Cenário main em B; no PR, C e D saem desse ponto — duas pontas até você integrar.

main        *─── B  ← ponta atual
                     |
feature/login  *─── D  ← ponta do branch
                  |
                  *─── C  feat: formulário
                  |
                  *─── (base em B)

Depois (commit M)

Efeito Nasce M só para fechar o PR — git show lista dois pais: B e D. O “caminho” C→D continua visível.

*   M  Merge branch 'feature/login' (#42)
|\          ↑ dois pais
| * D  fix: validação
| * C
* / B
|/
* A

Boa para times que querem “ver o PR” no grafo após integrar.
