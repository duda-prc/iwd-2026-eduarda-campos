Boas práticas 3/3 · Estratégias de merge na sua branch main

Rebase and merge (histórico linear)

> O que é: os commits do PR são reaplicados em cima da main (novos SHAs, ex.: C′ e D′) e a integração é um avanço em linha — sem nó M.
> Quando: prioridade em main quase reta, ferramentas de release/changelog que assumem poucos merge commits.
> Atenção: reescreve histórico — mesmo PR em outro clone precisa ser tratado com cuidado; combine com fluxo de “update branch” antes do merge.

Antes (feature à parte)

Cenário Commits C e D nasceram em cima de B, mas ainda não caminham “em cima” da ponta da main no repositório remoto.

main   *─── B  release 1.2
              |
feature *─── D  fix: edge case
          *─── C  feat: export CSV
          |
          *─── … ← ancora em B

Depois (linear na main)

Efeito Plataforma reaplica patches: C→C′, D→D′ — mesma ideia, commits novos SHAs.

main   *─── D′  fix: edge case
          *─── C′  feat: export CSV
          *─── B
          *─── A

Sem nó M; leitura do git log quase reta.
